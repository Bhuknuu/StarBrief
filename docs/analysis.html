01 / Overview

What Is StarBrief

The name combines A* search (the optimal-path algorithm) with "brief" for concise intelligence. StarBrief is a personal tool that solves a specific problem: relevant internships, hackathons, open-source programs, and research fellowships get posted across a dozen different platforms. Checking them manually means either wasting time every morning on repetitive searches or missing deadlines because you found the listing too late.

StarBrief automates the collection, cleaning, and prioritization. The output is a curated briefing, delivered through a dashboard, of opportunities ranked specifically against your interests and skills profile.

The system targets CS students in India, with a focus on AI, ML, NLP, and data science. The default profile in config/default_profile.json is configured for Developer, a CS student with primary interests in LLMs, computer vision, and generative AI.

03 / Architecture

System Architecture

The entire backend is stateless. There is no always-on server. Every compute job is a GitHub Actions cron that wakes up, does its work, writes to Neon PostgreSQL, and exits. The dashboard is a static Next.js app that reads from the database through serverless API routes.

Dual-Cadence Crawl Engine

The core innovation is splitting scraping into two cadences rather than running a full deep scrape every day. A full scrape every 24 hours would be slow, expensive in API calls, and likely to trigger rate limits. The dual cadence keeps daily runs under 15 minutes and weekly runs under 45 minutes.

Runs Daily at 2:00 AM IST
Daily Sentinel
  • Re-checks all active opportunity URLs with async HEAD requests
  • Marks expired, updated, or cancelled items in the database
  • Skims front pages and RSS feeds of each source
  • Queues new URLs into discovery_queue as breadcrumbs (URL + context only, no full scrape)
  • Writes a structured run log to scraper_runs
Budget: ~10-15 min / ~500 HTTP requests
Runs Sunday at 3:00 AM IST
Weekly Deep Crawler
  • Consumes all unprocessed breadcrumbs from the discovery queue
  • Paginates 3-5 pages deep into each enabled source
  • Hits Reddit API, university portals, and newsletter archives
  • Sends all new items through the full processing pipeline
  • Clears processed entries from the discovery queue
Budget: ~30-45 min / ~300 HTTP requests / ~100 LLM calls

Processing Pipeline

Every raw scraped item passes through five stages sequentially:

01
Rule-Based Pre-Filter
Keyword matching using exclusions.yaml. Must match at least one include keyword (AI, ML, intern, hackathon, open source, etc.) and must not match any exclude keyword (sales, marketing, HR, MBA, SSZONE Technologies). No API call. Eliminates roughly 60% of raw items before any cost is incurred.
02
Fuzzy and Semantic Deduplication
The thefuzz library compares title and organization strings. If both exceed 85% similarity with an existing record, the item is merged rather than inserted as a duplicate. For items that pass fuzzy dedup, cosine similarity of Google text-embedding-004 vectors catches semantic duplicates: the same internship posted on Internshala, LinkedIn, and the company site with different wording.
03
LLM Structuring with Reflection Loop
Groq (Llama 3 70B) receives a structured prompt and returns JSON: category, 2-3 line summary, skills array, year requirements, stipend range, geo scope, and base quality score. LiteLLM automatically reroutes to Gemini 1.5 Flash on any Groq error or timeout. A two-prompt Actor-Critic reflection loop then checks the output for hallucinated dates or URLs not present in the source text, triggering a re-extraction pass if the critic flags an issue.
04
Persona Scoring
A local Python function (no LLM) computes a relevance score between 0 and 1 against the user profile using four weighted factors: field match (40%), deadline urgency (25%), organization prestige (20%), and compensation (15%). The score maps to a priority bucket: Critical (0.75+), High (0.50+), Medium (0.25+), Low (below 0.25).
05
URL Validation and Write
Async HTTP HEAD requests verify that all apply URLs return 200. Valid items are bulk-inserted into opportunities. Items with invalid URLs are inserted with url_valid = false and status = 'unverified' rather than discarded, since the listing data is still useful. The discovery queue row is marked processed.

06 / Design System

Sanzo Wada Color System

The palette is derived from Sanzo Wada's 1933 "Dictionary of Color Combinations," a book of 348 curated color harmonies originally for print. The appeal is that these combinations were selected for timeless harmony rather than trend. On screen, they read as warm and distinctive instead of the cold SaaS blues that saturate the web.

The implementation derives tokens from Wada combinations #247, #268, #131, and #182, adapted for digital contrast requirements.

07 / Frontend

Dashboard Architecture

The dashboard is a Next.js 16 App Router application. Dependencies: React 19, TypeScript, Framer Motion for animations, CSS Modules for component-scoped styles. No Tailwind. No UI component library.

The Bookshelf Principle

One rule governs the layout: data lives in one canonical place. The sidebar is the single source of truth for all filter state. Both the Task View and Timeline View read from the same sidebar filter state. Switching views changes only the rendering, not the data query. Filter state is persisted to localStorage so it survives page refreshes.

Task View (Default)

A prioritized list grouped by deadline buckets: Today, This Week, This Month, Rolling (no deadline). Cards show a 3px priority-colored left border, organization, deadline countdown, category pill, and skills. Clicking a card expands an inline detail section. The Apply button appears on hover on desktop.

Timeline View

A Gantt-style calendar where each opportunity is a horizontal bar spanning from its discovery or event date to its deadline. Bar fill color encodes category. The left cap of the bar encodes priority. A dashed vertical line marks today. Bars have a minimum width of 40px to remain readable for single-day events. Rolling items appear in a separate row at the bottom.

10 / Notes

Technical Notes

CSS Utilities Without Tailwind

Tailwind is not installed. The dashboard uses CSS Modules for component-scoped styles and defines a small set of utility classes (flex, items-center, gap-4, etc.) in globals.css. These are not Tailwind classes; they are intentional custom utilities with the same names. Adding any new utility class requires defining it explicitly in globals.css.

Unstop: Prefer the API

Unstop has a public JSON endpoint at /api/public/opportunity/search-new. The implementation plan specifies using this over HTML scraping. This is strongly recommended. Unstop's HTML structure is complex and will break. The API response is stable.

Reflection Loop Cost Model

The two-prompt Actor-Critic pattern adds a second Groq call per item that passes through LLM structuring. At 50-100 items per weekly run, this is 100-200 total Groq calls per week. Both primary and fallback models are on free tiers. The current plan runs the critic on every item. A future optimization: run the critic only when the Actor output contains a future date or an external URL (the two most hallucination-prone fields), reducing calls by roughly 40%.

Internshala Rate Limiting

Internshala aggressively rate-limits scrapers. The sources.yaml configuration sets rate_limit_seconds: 3.0 and rotate_user_agent: true. Both are picked up automatically by BaseScraper.fetch(). The weekly deep scraper paginates 5 pages with these delays, meaning the Internshala scrape alone takes approximately 15 seconds minimum per page, or roughly 75 seconds total.

IIT / IISc Portal URLs

University summer program pages change URL structure every year when the new application cycle opens, typically in January or February. The URLs in sources.yaml for iit_iisc will need manual verification and updates at the start of each academic year.

Auto-Healing Scraper (Phase 2)

When a scraper returns zero results on a run where it previously returned 50 or more, the anomaly detector passes the raw HTML to Gemini 1.5 Flash (1M token context window) and asks it to identify updated CSS selectors. Those selectors are patched in-memory for the current run. This fires only on failure, costs exactly one Gemini call, and recovers a broken scraper without a code deployment.
